# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Cairn is a self-hosted, connected, day-reading wellness OS for training/nutrition/longevity with an agentic coaching loop. It is more than a tracker: it opens to a calm **Brief** that reads your whole picture and *suggests* what kind of day today should be, propagates flagged lab findings across every domain they touch (the "connected brain"), runs adaptive nutrition (MacroFactor-style expenditure + autoregulation feedback), learns who you are (about-me + family it plans around), captures effortlessly (frequents + voice + Apple Health), and surfaces quiet cross-domain insights one at a time. A single Node service serves four surfaces from one process (`src/server.ts`): a vanilla PWA (`/`), a REST API (`/api/*`), an MCP server (`/mcp`, Streamable HTTP), and a background scheduler. Storage is SQLite via Node's built-in `node:sqlite`. The product north-star/constitution — the "calm, suggestion-not-a-gate, no-scores, pull-never-push" rules every feature is held to — lives in `docs/VISION.md`; read it before shaping product behavior (the progress log there is maintained separately, do not edit it as part of code housekeeping).

## Commands

```bash
npm run dev      # tsx watch on src/server.ts -> http://localhost:8787 (auto-seeds on first boot)
npm run build    # tsc -> dist/
npm start        # node dist/server.js (run build first)
npm run seed     # seed default plan + profile if DB empty
npm run reset    # delete data/cairn.db* and re-seed
npm run migrate  # run src/migrate.ts manually (print current user_version, apply pending)
npm run backup   # curl /api/export -> cairn-export.json (server must be running)
docker compose up -d --build   # build/run locally on a Docker host
```

**Docker deploy.** The container bundles Node 24 and the coaching CLIs, so the host does not need
Node installed. Public releases are intended to publish a GHCR image plus a release
`docker-compose.yml`; see `docs/SHARING.md`. Migration/restore internals live in
`docs/OPERATIONS.md`.

**Tests:** `npm test` runs a zero-dependency `node:test` harness (`test/**`) — `pretest` builds `dist/`, `test/run.mjs` points `DATA_DIR`/`DB_PATH` at a throwaway temp dir per run (never touches real `data/cairn.db`), and the suite is deterministic + offline (no agent CLI, no network). It covers the constitution-critical deterministic cores (`extractJson`, `dayRead`, `estimateExpenditure`, marker trend/forecast, `deriveDirectives`/clusters/the vitamin-D side-guard, self-updating memory dedup/supersession, `verifyCitation`/`safetyGate`/`isPlausibleSourceUrl`, the apply-path clamp guardrails, settings round-trip, the migration ladder) including a GOLDEN test that the serialized markers never leak `impact_score`. `npm run lint` runs Biome lint rules only; `npm run format` formats in place and should stay scoped to touched files. The `stub` agent (in `agents.json`) is the offline smoke-test path for the propose/apply loop — it needs no API key and returns a canned JSON proposal.

## Hard requirements & conventions

- **Node 24 is mandatory** — that's where `node:sqlite` is unflagged. No native build step.
- **ESM with explicit `.js` import extensions** (`tsconfig` is `NodeNext`). When importing a sibling `.ts` module, write `from "./repo.js"`, not `"./repo"`. Match this in every new file.
- The DB is a single shared `DatabaseSync` instance exported from `src/db.ts`; all schema lives there as `CREATE TABLE IF NOT EXISTS`. For a **brand-new table**, add a `CREATE TABLE IF NOT EXISTS` in `db.ts`. To add a **column to an existing table**: (1) add it to the matching `CREATE TABLE IF NOT EXISTS` in `db.ts` so fresh DBs get it, and (2) append a new entry to the `MIGRATIONS` array in `src/migrate.ts` with the next integer `version` and an idempotent `up(db)` that does the `ALTER TABLE … ADD COLUMN` inside a try/catch (so it's a no-op on DBs that already have the column). `runMigrations(db)` is called automatically at the bottom of `src/db.ts` on every boot; it reads `PRAGMA user_version`, applies any pending migrations in order, then updates `user_version`. Down-migrations are not supported — back up before deploying schema changes. You can also run `npm run migrate` manually. See `docs/OPERATIONS.md` for the full deploy/migration/backup playbook. **The schema is at `user_version` 47** (`sw.js` `cairn-v154`). A **self-hosted update-detection** round (migration **v47** `settings-update-check-enabled`; `sw.js` `cairn-v154`) lets an instance detect a newer release and GUIDE the update (never auto-apply — a container can't safely replace its own image, mirroring the opt-in `cairn-update-agent-clis` stance): `src/version.ts` resolves the running version (`CAIRN_VERSION` env baked from the release tag → `package.json` → `0.0.0`) with a dependency-free SemVer comparator (`parseVersion`/`compareVersions`/`isNewer` — a release outranks its pre-release, garbage never reads as newer), and `src/updateCheck.ts` checks the **GitHub Releases API** for the latest tag, caching `{latest,html_url,notes,checked_at,error}` in the `app_state` KV (pure `parseLatestRelease`/`computeUpdateStatus`/`getUpdateStatus`/`checkForUpdate` — the network call never throws, folding errors into `status.error`). ## Architecture: two protocol surfaces over one repo layer

The most important structural fact: **`src/api.ts` (REST) and `src/mcp.ts` (MCP) are both thin adapters over the same `src/repo.ts`.** They expose largely parallel operations. Any business-logic change belongs in `repo.ts` (or `prompt.ts`); the two surfaces should stay near-mirror wrappers. When you add a capability, expect to touch both `api.ts` and `mcp.ts` plus the underlying `repo.ts` function. Multi-step **agentic** operations that both surfaces run (session-suggest, nutrition check-in, meal swap, recipe, health review, insight generate) live in **`src/coachOps.ts`** — each exports one function returning a plain `{ok, …}` result; `api.ts` wraps it in `res.json(...)`, `mcp.ts` in `asText(...)`. The shared `runChosen(agent, prompt)` (explicit agent, or `"auto"`/blank → configured rotation with fallthrough) also lives there. Put new agentic orchestration in `coachOps.ts`, not duplicated in the adapters.

- `src/repo.ts` — a thin **barrel** that re-exports the domain modules under `src/repo/` (`exercises`, `plan`, `sessions`, `profile`, `activities`, `memory`, `nutrition`, `chat`, `settings`, `health`, `coach`, `propagation`, `evidence`, `intelligence`, + `shared`); external code keeps importing everything from `./repo.js` unchanged. - `src/chatTurns.ts` — **durable, token-streaming chat-turn engine** (the conversational counterpart to `enrich.ts`). Chat is no longer a blocking request/response: `POST /api/chat` persists the user message + a `chat_turns` row (status `queued`) and hands the id to this **serial in-process worker** (one CLI agent at a time), returning immediately. `runChatCompletion` runs the turn under an `AbortController`: the first agent in the (explicit or rotation) order **STREAMS** if it's stream-capable (`runAgentStreaming` → emits live `delta` events on the bus, sentinel-aware so the trailing actions JSON never reaches the bubble); any failure or a non-streaming agent falls back to a **one-shot rotation** (chat's success criterion is NON-EMPTY TEXT, not parseable JSON, so it deliberately does NOT reuse the JSON-centric `runAgentWithFallback`). Progress rides an **`EventEmitter` bus** (`onTurnEvent(id, cb)`) the SSE endpoint forwards: `phase` / `delta` (a live reply chunk) / `reset` (streaming fell back — clear the partial bubble) / terminal `done`/`error`/`canceled`. `enqueueChatTurn`/`cancelTurn`/`recoverChatTurns` are the lifecycle entries; `cancelTurn` flips state then SIGKILLs the live subprocess (a Stop — `agents.ts` `runAgent`/`runAgentStreaming`/`runAgentWithFallback`/`runChosen` thread an `AbortSignal`). Degrades like the rest of the loop (no agent → a calm error reply persists, nothing throws). Because the turn lives in SQLite, a follow-up queued mid-think — or a turn interrupted by a tab switch / reload / restart — survives: `recoverChatTurns()` (boot, alongside `recoverPendingEnrich`) marks interrupted `running` turns `error` (their actions may have partially applied, so re-running risks duplicates) with a calm thread note and re-enqueues `queued` ones.
- **Headless token streaming** (`agents.ts`): `AgentDef.stream?: { format, args }` declares a CLI's NDJSON streaming mode. `runAgentStreaming` spawns with those args, reads stdout LINE BY LINE, and maps each event to assistant text via `streamDelta(format, line)` — a CONSERVATIVE per-CLI adapter (unknown shape → null, never garbage → caller falls back). Verified support: **Claude** (`-p --output-format stream-json --include-partial-messages`, true token deltas — verified live), **Grok** (`--output-format streaming-json`, emits `{"type":"text","data":"…"}` text deltas + `{"type":"thought",…}` reasoning to skip — verified live on grok 0.2.54; the older xAI ACP `agent_message_chunk` shape is still tolerated as a fallback). **Codex** delivers its agent message only as a complete `item.completed` event (no token deltas) and **Antigravity** (`agy`) has no streaming flag, so both stay one-shot (as do non-configured agents + `stub`). `agentSupportsStream(name)` gates the path. The **reply-marked chat contract** (`prompt.ts` `buildChatPrompt` + `CHAT_REPLY_SENTINEL`/`CHAT_ACTION_SENTINEL` + `parseChatReply`) is what makes streaming clean: the model writes a `===CAIRN_REPLY===` line, then its athlete-facing reply as plain prose (streamed verbatim), then — only if it needs to act — a `===CAIRN_ACTIONS===` line followed by `{"actions":[…]}`. `parseChatReply` drops everything **before** the reply marker (autonomous agents like the gemini/antigravity CLI narrate their tool steps as text first — "I will query the … table" — which must never reach the bubble) and splits the actions on the action sentinel; the streaming `flush` in `chatTurns.ts` is marker-aware (it withholds a forming sentinel at the tail and fires a `reset` to wipe any pre-marker narration the moment the reply marker lands). A conservative `stripLeadingNarration` safety net covers agents that ignore the marker — it drops only leading lines carrying BOTH a step verb AND a filesystem/db token, so genuine coaching prose ("Let me explain your zones", "I'll bump your squat") is never touched. Backward-compatible with a marker-less reply and the legacy `{reply, actions}` JSON alike (covered by `test/streaming.test.js`).
- `src/art.ts` — **generated artwork service**: photoreal/stylized images for foods, exercises and activities via Google's `gemini-3.1-flash-image` (override with `GEMINI_IMAGE_MODEL`; needs `GEMINI_API_KEY`), cached in `data/art/` as `sha1(kind:normalized text).png`. A strictly serial in-process queue (mirrors the enrich.ts pattern: in-flight dedup by cache key, a throwing job never breaks the drain loop, plus an in-memory negative cache so failed keys aren't retried until restart) generates on cache miss while `GET /api/art` returns 204 immediately; it's a **direct REST call** (global fetch, 60s AbortController timeout, atomic tmp-then-rename write) — NOT an agents.json CLI run. Degrades gracefully: no key / `settings.art_enabled` off / known-failed → 204, nothing runs. **Semantic cache layer**: before any image call, `resolveConcept()` runs one cheap Gemini *text* call (`GEMINI_TEXT_MODEL`, default `gemini-3.1-flash`, 20s timeout, temperature 0) asking whether the query would render essentially the same picture as an existing `art_assets` row (comparison window: 150 most recent per kind; strict for exercise/activity — different movement/equipment is not a match; lenient for food — same dish reworded is) and, if not, what canonical phrase to file the new image under; the verdict persists in `art_aliases` (normalized query → asset key), so each unique phrase pays at most one text call ever and `cachedArtPath`/`requestArt` resolve through aliases on every later hit. Any canonicalize failure falls back to generating under the query's own key. **Spend telemetry**: every paid call (and every avoided generation) is recorded via `repo.recordArtUsage` into `art_usage` — actions `generate` (flat `ART_IMAGE_COST_USD`, default $0.039/image), `canonicalize` (token-priced via `usageMetadata` at `ART_TEXT_IN_USD_PER_M`/`ART_TEXT_OUT_USD_PER_M`), `reuse` (carries `est_saved_usd`), `fail`; surfaced by `GET /api/art/stats` / MCP `get_art_stats` with a since-`art_enabled_at` window plus all-time. `warmArt()` pre-queues every image the PWA will ask for (exercise names; the current non-discarded meal plan + draft's meals; ~30 recent food notes; distinct recent activity types via a duplicated `ACT_ART_PHRASE` map that MUST stay in sync with `public/app.js`) — queries are built exactly like the PWA's (`.trim().slice(0,120)`) so cache keys match; it runs ~5s after boot in `server.ts` and on `POST /api/art/warm`, returning `{queued, skipped}` (`requestArt` makes it a safe no-op when generation is unavailable). `data/art/` is a regenerable cache (safe to exclude from backups). Set `GEMINI_API_KEY` in the container environment, `.env`, or Settings if generated artwork should run; otherwise the PWA stays on SVG fallback. An **offline seed-art pack** (`seed-art/` — pre-baked, ffmpeg-downscaled images keyed by the SAME `cacheKey` as the live cache, + a `manifest.json`) ships in the repo so a fresh seed / the demo renders real studio photos with NO key at runtime: `installSeedArt()` copies the files matching THIS DB's `enumeratePwaArt()` queries into `data/art/` (idempotent — never clobbers a real image, no-op when the pack is absent in a slim checkout), wired into `seedIfEmpty()` (`seed.ts`) + the demo CLI (`demoSeed.ts`, now exporting `seedDemo()`). Rebuild it rarely (after changing demo content / art style) with `npm run seed:art:build` (`src/buildSeedArt.ts` — needs a Gemini key + ffmpeg; seeds the demo into a THROWAWAY DB, `pregenerate`s each image under its DIRECT cache key bypassing semantic-dedup so the pack is alias-free, downscales to a small JPEG, writes the pack + manifest, prunes orphans). The `GET /api/art` route sniffs the magic bytes (`0xFF 0xD8` → `image/jpeg`) so a JPEG-in-`.png` pack file serves with the honest type under `nosniff`.

## The agentic coaching loop (propose → review → apply)

Coaching never auto-applies. The flow:

1. `src/prompt.ts` builds a prompt embedding the full `getCoachContext()` snapshot and a strict JSON output contract (plus non-negotiable training/nutrition guardrails — conservative load progression, injury `constraint_note`s, lean-safe deficits). `buildMealPlanPrompt` is framed as a longevity/getting-lean coach: a shared `LONGEVITY_GUARDRAILS` block (protein anchor ~0.7–1 g/lb, 30g+ fiber, whole foods, oily fish 2–3×/wk, lean-safe only), flexible meal-slot timing rules (no pre-workout meal for fasted morning training — substantial post-training breakfast instead), the training split, and `settings.meal_prefs` verbatim. `buildMealSwapPrompt({plan, day, mealIndex, hint})` drives the single-meal swap (replacement within ±10% kcal/protein unless the hint overrides; bare single-meal JSON contract). **Vision-build prompt builders, all bound to the constitution** (calm, suggestion-not-a-gate, no 0-100 scores, pull-never-push, health findings are informational not medical advice): `buildDayReadPrompt` (the Brief — reads the whole picture, honors an athlete override, protects earned rest), `buildSessionPrompt` (on-demand session honoring time/equipment/injury + the day read), `buildNutritionCheckinPrompt` (adherence-neutral adaptive check-in over the derived expenditure — proposes a small ±100–250 kcal change only on genuine drift, protein-first floors, `change:false` the common answer), `buildInsightPrompt` (ONE genuine cross-domain connection or `{found:false}`, deduped against `recentInsightTexts`) and `buildWeeklyReadPrompt` (a standing "how the week went + the one change" stored as a `weekly_read` insight). **The connected brain is rendered into every plan-shaping prompt** by the shared `renderConnectedBrain(ctx, {domains})` helper, which folds the active cross-domain `directives` (filtered to the relevant domain — nutrition for meals, training/watch for the coach/day-read) plus the unified `recovery` view into a compact plain-language block, so a flagged lab already shapes meals & training (returns `""` when there's nothing to say). `CONTEXT_GUARDRAILS` (shared across the coach/chat/meal prompts) now also covers FAMILY (plan around commitments, let `about_me` personalize tone) and HEALTH DIRECTIVES (fold nutrition/training directives in, respect `watch` items; an `uncertain`/uncited directive is a softer nudge). `buildHealthReviewPrompt`'s `HEALTH_REVIEW_SCHEMA` now emits a `directives[]` array (persisted via `applyReviewDirectives`). `buildChatPrompt` adds a **PROGRESSIVE UNDERSTANDING** guardrail: when the DATA shows an obvious gap (no `about_me`, an unknown training-time / food like-dislike) the agent MAY ask ONE brief low-friction question and emit an `add_memory` action for the answer — never a questionnaire. Chat safe-actions (applied immediately) now also include `log_food`, `log_health` (extracts every marker into Health records) and `add_context_event` (trips/injuries/life events/family commitments).
2. `src/agents.ts` `runAgent()` spawns an **external CLI subprocess** (Claude Code, Codex, Antigravity, Grok, or `stub`) defined in `agents.json`, substituting `{prompt}` into the args. `extractJson()` pulls the JSON object out of the CLI's stdout.
3. The result is stored as a `draft` row (`plan_proposals` or `meal_plans`) — never applied directly.
4. `applyProposal()` walks `parsed.changes[]` and calls `updateTarget()` per change, flipping status to `applied`. (It also recognizes two newer shapes: a `parsed.days` payload routes to `replacePlan()` for a full restructure, and a `kind:'nutrition_target'` payload from the adaptive nutrition check-in is advisory — there's no plan to mutate, so "applying" it is a clean acknowledgement on every surface, surfaced via the Energy Balance card, not the plan-proposals apply button.)

**Agent selection.** Callers either name an agent or pass `agent: "auto"` / omit it. Auto uses `repo.pickAgentOrder()` (strategy from the `settings` table: `round_robin` default / `random` / `priority`, over the enabled agents in their configured order) and `agents.runAgentWithFallback()`, which tries agents in order until one returns parseable JSON — so a dead CLI login or timeout transparently falls through to the next. `api.ts` and `mcp.ts` share a `runChosen()` helper for this. `agents.json` still defines each backend (command, args, stdin-vs-arg input, env requirements); which are **enabled/ordered** and the **strategy** live in `settings` and are editable at runtime (PWA Settings tab, `/api/settings`, or `get_settings`/`set_settings` MCP tools) — no redeploy.

`src/scheduler.ts` is an optional weekly auto-draft: a 60s `setInterval` reads `settings` each tick (so the schedule is toggleable live) and, when `coach_enabled` and the configured `coach_day`/`coach_hour` match, drafts one proposal via the auto rotation and leaves it for manual apply. First-run settings seed from the legacy `COACH_AGENT`/`COACH_DAY`/`COACH_HOUR` env vars.

**Conversational chat** (`POST /api/chat`, `buildChatPrompt`) is a **prose-first, streaming** loop: the agent writes the reply as plain prose (rendered live, token by token when the agent streams — Claude/Grok), then — only to log or change something — a `===CAIRN_ACTIONS===` line + `{"actions":[…]}` (`parseChatReply` splits them; see `src/chatTurns.ts` + the headless-streaming bullet above). Safe actions apply immediately (`log_activity`/`log_set`/`set_profile`/`add_memory`/…) while plan-affecting actions become **draft proposals** — `plan_update` (target tweaks → `parsed.changes`) and `plan_restructure` (frequency/split change → `parsed.days`); `applyProposal()` handles both (a `parsed.days` payload routes to `replacePlan()`, otherwise it loops `updateTarget()` over `parsed.changes`). **Chat is now a durable, non-blocking TURN** (see `src/chatTurns.ts`): `POST /api/chat` returns `{ok, turn, user_message}` at once after enqueuing — it never holds the request open while the agent runs. The PWA streams real progress over **`GET /api/chat/turns/:id/stream`** (Server-Sent Events: a `snapshot` then `phase`/`done`/`error`/`canceled` events; reached with `?token=` under auth since EventSource can't set headers), reconstructs the in-flight + queued thread from **`GET /api/chat/turns`** (active = queued+running, oldest-first) on every (re)load, polls **`GET /api/chat/turns/:id`** as an SSE fallback, and stops a turn with **`POST /api/chat/turns/:id/cancel`**. So a follow-up typed while the coach is thinking just queues (its own turn, drained serially), and neither it nor an in-flight turn disappears across a tab switch / reload / restart. Completed turns still persist in `chat_messages`. The Chat tab groups messages under day dividers with older days collapsed behind a "Show earlier" affordance, offers starter chips when the conversation is empty, and has a header "fresh start" button → `POST /api/chat/reset` (distill-to-memory, then archive).

## Background enrichment (`src/enrich.ts`)

Free-text activity logs and food notes save **instantly** via the offline regex parser (`parseActivity`), then — if `settings.enrich_enabled` (default on) and an agent is reachable — a **background, in-process serial queue** runs `buildEnrichPrompt()` through `runAgentWithFallback()` to (a) refine the entry's structured fields and (b) distill genuinely-notable durable facts into the `memory` table. **Uploaded health documents** (bloodwork/DEXA/other) are the third enrichment kind: the job hands the agent the **absolute file path** (`buildHealthEnrichPrompt`) — the CLIs (Claude Code/Codex) can open local files — to extract markers + a summary + memory. It degrades gracefully end-to-end: disabled / no agent → `skipped` (written directly at insert, no queue churn); a failure or wrong-shape (e.g. coach-proposal) response → `failed`/no-op with the regex parse (or the as-uploaded doc) left intact.

- **Kinds**: `'activity' | 'food' | 'health'`. **Status machine** on `activities`/`food_notes`/`health_documents` `.enrichment_status`: `pending` → `in_progress` (set *before* the agent `await`, so a crash leaves a recoverable marker) → `done`/`failed`/`skipped` (or `null` = not applicable). `recoverPendingEnrich()` (called from `server.ts` at boot) re-enqueues `pending`+`in_progress` rows for all three kinds.
- The queue is **strictly serial** (one CLI subprocess at a time), uses a shorter 120s timeout than the coach loop, and a throwing job never escapes the drain loop. A health job refuses a non-absolute `file_path` (skips) — that's the only thing constraining the agent's file read to uploads.
- Agent-provided fields are **type-coerced and clamped** before write; `memory` items are added via `addMemory` (which dedupes exact repeats). Prompt-only dedup is a soft hint, so the code-level dedup is the real guard.
- The frontend **polls** `GET /api/activities/:id` / `/food-notes/:id` / `/health-docs/:id` to watch the status flip and upgrade the entry in place.
- **Health ingestion is completeness-first** (a modern panel like Function Health lists 100+ markers; a weaker model used to curate that down to "the interesting ones"). Three deterministic guards make it transcribe EVERY marker: (1) the health job picks `repo.pickHealthAgentOrder()` — Claude-first, then Codex, then the rest, NO round-robin cursor side effect, an explicit `agent_routes.health` pin still wins — instead of the load-spreading rotation, because faithful transcription matters more than spreading load; (2) `buildHealthIngestPrompt` carries a hard "transcribe every marker verbatim, do not curate, include the in-range/normal long tail (full CBC differential, electrolytes, the whole urinalysis, omega sub-fractions, every hormone, blood type)" contract + a self-reported per-panel `marker_count` (the old prompt literally said "prefer real, decision-useful markers" — the curation bug); (3) for a **pasted-text** source (`text/plain`) the job estimates the source's marker count with `repo.estimateMarkerCandidates(text)` (counts value lines, excludes section headers + flag words) and, if the extraction is grossly short (`< 80%` of an estimate ≥ 40), re-runs ONCE Claude-first with an `emphasizeCompleteness` nudge and keeps whichever attempt captured more. The per-panel cap is `repo.MAX_MARKERS_PER_PANEL` (250, was a silent 100). The chat `log_health` action carries the same "no curation" instruction and points big pastes at the Health tab's paste box for the more reliable path.

**Personal-context layer.** `health_documents` (uploaded labs/scans → analyzed markers + summary) and `context_events` (trips / injuries / life events, with dates + a `meta_json` for kind-specific fields) both feed `getCoachContext()`. A shared `CONTEXT_GUARDRAILS` block in `buildCoachPrompt`/`buildChatPrompt`/`buildMealPlanPrompt` directs the coach to plan **around** them: travel-friendly/deload weeks over trip dates, de-load/swap exercises hitting an injured area, dial volume back during flagged life events, and factor health flags (e.g. low ferritin) — explicitly framed as informational, not medical advice.

## Domain gotchas

- **Weight encoding**: negative `weight`/`target_weight` = assisted movement (e.g. `-30` = 30 lb assist); `null` = bodyweight. The PWA and prompts rely on this.
- **Timed exercises** (`exercises.mode = 'timed'`): sets log `duration_sec` (weight/reps may be null), plans prescribe `target_seconds`, and progression is in seconds — never load. Proposal `changes[]` may carry `target_seconds` instead of `target_weight`.
- Est-1RM is Epley on the best set per day.
- Sessions are keyed by date — `getOrCreateSession` reuses today's session, so logged sets accumulate into one session per day.
- **Garmin strength reconciliation.** A synced Garmin *strength* activity (`isStrengthGarminType`) is modeled as a Cairn **session**, not a loose activity — `upsertGarminActivity` deliberately skips the generic `activities` row for it (no RECENT duplicate). `src/garmin.ts` also pulls Garmin's detected `exerciseSets` (`fetchExerciseSets`, bounded by `GARMIN_STRENGTH_LIMIT`). After each sync (and via `POST /api/garmin/reconcile` / MCP `reconcile_garmin_strength`), `repo.reconcileGarminStrength()` runs the **deterministic** merge — attaches the physiology blob (HR/zones/calories/TE) to `sessions.garmin_json`, links `garmin_activities.session_id`, deletes any stale generic row — *always*, even with no agent. Then the **agentic** layer runs on the serial enrichment queue (`enrich.ts` kind `'garmin_strength'` → `buildGarminStrengthPrompt`): it writes the one-line "body's reaction" summary and **logs the exercises Garmin detected that aren't already in the session** via `logSetByName` (a code-level guard skips any exercise that already has logged sets, so hand-logged sets are never overwritten/duplicated and re-syncs stay idempotent). Degrades cleanly: enrichment off / no agent → physiology merges, no narrative, no extrapolation. Surfaced as the "Garmin · body's reaction" card on Today (`garminSessionCard`).
- **Autoregulation feedback** (`sessions.soreness`/`performance`/`joint_pain`, set via `setSessionFeedback`) is optional 1-tap signal that INFORMS coach selection (bend volume, de-load a sore joint) — it NEVER overrides progressive overload or auto-changes the plan. Likewise the subjective `checkins` are a parallel day-read signal and the graceful-degradation path when there's no wearable; absent signals never force a rest read.
- **The connected brain (markers → directives propagation).** A flagged/sub-optimal lab marker propagates into the domains it touches (nutrition/training/watch) as `health_directives` via the deterministic `deriveDirectives()` engine and `MARKER_MAPPINGS`. Two sources coexist and each clears/rewrites only its own rows: `'markers'` (the deterministic engine) and `'health_review'` (agent-emitted on a saved review). Directives are INFORMATIONAL, not medical advice; a directive flagged `uncertain` (or lacking a `citation`) is a softer nudge, not a hard rule. Nothing auto-applies — the user flips `active|resolved|dismissed`.
- **Optimal-zone vs lab-range.** `OPTIMAL_ZONES` are evidence-anchored longevity/preventive-cardiology target bands (ApoB, LDL, HbA1c, ferritin, vitamin D, …), DISTINCT from the lab's population reference interval — a value can read "in range" yet sit far from optimal. `prioritizeMarkers()` ranks by an INTERNAL `impact_score` (distance-from-optimal, weighted by whether there's an actionable lever, plus a flag floor); **never surface that score** — the constitution bans 0-100 grades. The UI shows in/out-of-optimal and the direction, never the number. Markers are also bucketed into meaningful health groups (`MARKER_GROUPS`, longest-match-wins on the name, same matcher shape as `matchOptimalZone`) and carry a deterministic `trend` (`{dir:'rising'|'falling'|'stable'|null, change, span_days, n}` — stable when the net change is small vs the series' own spread); `getMarkerHistory()` and `prioritizeMarkers()` both return `{ markers, groups }` (canonical-ordered `groups` present) with per-marker `group`/`group_label`/`trend`. The review prompt surfaces each priority marker's group + trend direction + reading recency so the coach speaks to trajectory ("rising over ~14 mo"), not just the latest value. Reading recency renders human via `relAge()`/`absDate()` (see `docs/DESIGN.md`), never a bare month-year.
- **Product constraints (from `docs/VISION.md`, enforced in prompts AND UI).** The Brief and every read are a SUGGESTION, never a gate or verdict — the athlete drives. NO numeric scores anywhere (no 0-100 grades, no metric walls). Insights and the weekly read are PULL, never push — they wait in-app, never a notification or nag; the generator emits ONE genuine connection or `{found:false}` and dedupes against what it already said. Adaptive nutrition is adherence-NEUTRAL: a thin logging week lowers confidence, never blames; `change:false` is the calm, common answer.
- **No authentication by default** (single-user, trusted-network model). An OPTIONAL shared token gates `/api` and `/mcp` when `CAIRN_AUTH_TOKEN` is set — implemented as one global guard in `src/auth.ts` (wired in `src/server.ts` before the routers; exempts `GET /api/health` and the static PWA shell). Unset = behaves exactly as before. The PWA sends it as `X-Cairn-Token` (or `?token=` on direct resource URLs via the `withToken()` helper); MCP/API clients use `Authorization: Bearer`. Keep it on localhost, a LAN, Tailscale/VPN, or another trusted private network regardless. The `cairn-home` Docker volume holds live OAuth tokens for the CLI logins. `src/server.ts` also sets conservative security headers (CSP/nosniff/frame-deny, `x-powered-by` off) and scopes the 25mb body limit to the health-doc upload route only (1mb elsewhere). The container runs its main process as a non-root `app` user (an entrypoint fixes volume ownership then drops privileges); one-off `docker compose exec` commands that must persist a login need `-u app`. An OPTIONAL per-IP **rate limiter** (`rateLimitGuard` in `src/auth.ts`, wired in `src/server.ts` *before* `authGuard` so it also blunts token brute-force) activates only when auth is on and `CAIRN_RATE_LIMIT > 0` (defaults: 600 req / 60s; `CAIRN_RATE_WINDOW_MS` tunes the window) — a calm defense-in-depth cap, same `/api`+`/mcp` scope as auth, `GET /api/health` exempt; the decision is a pure, unit-tested `rateLimitDecision`.

## PWA

`public/` is dependency-free vanilla JS. The tabbed client (Today / Plan / Progress / Chat / Me / Settings) that talks only to `/api/*` lives in **`public/js/*.js`** — split from the former single `app.js` into ten ordered modules (`01-core` … `10-boot`), loaded as classic `<script>`s that **share one global scope** (no bundler; load order matters, and `10-boot.js` runs the boot sequence last — so a top-level eager cross-module reference must use a lazy `() => fn()` thunk, since function hoisting does NOT cross `<script>` boundaries; see `ME_HANDLERS`/`PROGRESS_HANDLERS`). `sw.js` is a minimal cache-first service worker — **any change to files under `public/` MUST bump the `CACHE` version constant at the top of `sw.js` in the same commit** (currently `cairn-v126`; the precache `ASSETS` list must include any new static asset — e.g. the vendored `/vendor/xterm*` for the agent-login terminal), or installed PWA clients keep serving the stale assets forever. `sw.js` now **`skipWaiting()`s on install** and the client **reloads once on `controllerchange`** (in `10-boot.js`, guarded against the first-ever install), so a deploy goes live on the next open — no manual "tap to refresh", no stale-cache limbo (a client once silently fell ~40 versions behind the old wait-for-tap model); the version bump is still mandatory, since that's what makes the browser fetch + install the new worker at all. No build step touches `public/`. The visual language is the **"Atelier" design system** — warm-cream editorial theme (Fraunces display + Schibsted Grotesk UI, terracotta/sage accents) — whose frozen contract (palette, class names, motion rules) lives in `docs/DESIGN.md`; read it before touching `styles.css` or view markup. `public/art.js` is the **CairnArt SVG illustration library** (plain script loaded before the `js/` modules): `CairnArt.food(text)` / `.exercise(name, muscleGroup)` / `.activity(type)` keyword-map inputs to hand-drawn studio-style illustrations and return static SVG strings — caller text is never interpolated into the SVG, so its output is inserted into `innerHTML` raw (everything else still goes through `escHtml`/`escAttr`); the client wraps every call in the guarded `art()` helper so a missing art.js degrades to text-only. a History session is **tap-to-edit** (correct logged set numbers + notes via the `openSessionEdit` overlay). **Me** has a Profile / Memory / Health / Life / **Family** sub-nav: Profile now includes the free-text **about_me** field (`PUT /api/profile`), Memory is the curate-able coach memory, **Health** keeps the uploaded-docs + **marker-trends** views (markers grouped into meaningful health groups — Lipids & Cardiovascular, Metabolic & Glucose, Iron & Red Blood, … — each row tap-expands into an inline progress chart with the optimal-zone band shaded, the trend in plain words, and human-friendly recency like "3 months ago"; the tab is fed by `/api/markers/priority`, the optimal-band superset) AND a **Brain** view (recovery from `/api/recovery`, optimal-zone **priority markers** from `/api/markers/priority`, and grouped **directives** from `/api/directives` with status flips and a re-derive button → `/api/directives/derive`), **Life** is the trips/injuries/events timeline (with a subtle active-context banner mirrored on Today), and **Family** is the CRUD roster (`/api/family`). **Settings** adds an **Apple Health** connect card (the iOS Shortcut posts daily metrics to `/api/health-metrics`). **Chat** is a **dynamic, non-blocking, token-streaming** surface: the composer never locks, so a follow-up typed while the coach is thinking just **queues** (rendered as a "Queued" bubble); a single `EventSource` streams the oldest active turn (`chatMonitorEnsure`/`chatOpenStream`, finalizing each bubble **in place** so order holds with several queued), advancing to the next as each finishes. When the chosen agent streams (Claude/Grok), the reply renders **token by token** — a `delta` event appends to a live `.stream-text` span with a blinking `.stream-caret` (`appendStreamDelta`/`ensureStreamingBubble`), then `finalizeTurn` swaps in the full markdown; a `reset` event reverts to the calm "Thinking…" dots (streaming fell back to one-shot, or the reply marker arrived and any pre-reply tool narration is being wiped). Each pending/streaming bubble carries a **Stop** control (`cancelTurn` → `/api/chat/turns/:id/cancel`); the unsent **draft** is mirrored to `localStorage` (`saveChatDraft`/`loadChatDraft`) and the in-flight + queued thread is rebuilt from `/api/chat/turns` (`chatReconnect`) on every render — so nothing disappears across a tab switch / reload / restart. The monitor tears down when leaving the Chat tab (`chatTeardownMonitor` in `switchTab`/`activateTab`). Enrichment polling guards against stale-tab writes via a `pollToken` bumped on every (re-)render. All server-supplied strings rendered into `innerHTML` must go through `escHtml`/`escAttr`.
