# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Cairn is a self-hosted, connected, day-reading wellness OS for training/nutrition/longevity with an agentic coaching loop. It is more than a tracker: it opens to a calm **Brief** that reads your whole picture and *suggests* what kind of day today should be, propagates flagged lab findings across every domain they touch (the "connected brain"), runs adaptive nutrition (MacroFactor-style expenditure + autoregulation feedback), learns who you are (about-me + family it plans around), captures effortlessly (frequents + voice + Apple Health), and surfaces quiet cross-domain insights one at a time. A single Node service serves four surfaces from one process (`src/server.ts`): a vanilla PWA (`/`), a REST API (`/api/*`), an MCP server (`/mcp`, Streamable HTTP), and a background scheduler. Storage is SQLite via Node's built-in `node:sqlite`. The product north-star/constitution — the "calm, suggestion-not-a-gate, no-scores, pull-never-push" rules every feature is held to — lives in `docs/VISION.md`; read it before shaping product behavior (the progress log there is maintained separately, do not edit it as part of code housekeeping).

## Commands

```bash
npm run dev      # tsx watch on src/server.ts -> http://localhost:8787 (auto-seeds on first boot)
npm run build    # tsc -> dist/
npm start        # node dist/server.js (run build first)
npm run seed     # seed default plan + profile if DB empty
npm run reset    # delete data/cairn.db* and re-seed
npm run migrate  # run src/migrate.ts manually (print current user_version, apply pending)
npm run backup   # curl /api/export -> cairn-export.json (server must be running)
docker compose up -d --build   # build/run locally on a Docker host
```

**Docker deploy.** The container bundles Node 24 and the coaching CLIs, so the host does not need
Node installed. Public releases are intended to publish a GHCR image plus a release
`docker-compose.yml`; see `docs/SHARING.md`. Migration/restore internals live in
`docs/OPERATIONS.md`.

There is no test suite, linter, or formatter configured. The `stub` agent (in `agents.json`) is the offline smoke-test path for the propose/apply loop — it needs no API key and returns a canned JSON proposal.

## Hard requirements & conventions

- **Node 24 is mandatory** — that's where `node:sqlite` is unflagged. No native build step.
- **ESM with explicit `.js` import extensions** (`tsconfig` is `NodeNext`). When importing a sibling `.ts` module, write `from "./repo.js"`, not `"./repo"`. Match this in every new file.
- The DB is a single shared `DatabaseSync` instance exported from `src/db.ts`; all schema lives there as `CREATE TABLE IF NOT EXISTS`. For a **brand-new table**, add a `CREATE TABLE IF NOT EXISTS` in `db.ts`. To add a **column to an existing table**: (1) add it to the matching `CREATE TABLE IF NOT EXISTS` in `db.ts` so fresh DBs get it, and (2) append a new entry to the `MIGRATIONS` array in `src/migrate.ts` with the next integer `version` and an idempotent `up(db)` that does the `ALTER TABLE … ADD COLUMN` inside a try/catch (so it's a no-op on DBs that already have the column). `runMigrations(db)` is called automatically at the bottom of `src/db.ts` on every boot; it reads `PRAGMA user_version`, applies any pending migrations in order, then updates `user_version`. Down-migrations are not supported — back up before deploying schema changes. You can also run `npm run migrate` manually. See `docs/OPERATIONS.md` for the full deploy/migration/backup playbook. **The schema is at `user_version` 27.** Migration **v27** (`day-read-override`) adds `day_reads.override` — the athlete's persisted Brief steer ("rough night" / "give me an easy day" / …). The day-read is no longer a throwaway client-only reshape: `computeDayRead` (`src/dayread.ts`) always persists the read (steer included) via `repo.saveDayRead`, which has a no-clobber guard so a canonical/nightly recompute never wipes a steer; `GET /api/today-read?reset=1` invalidates a steer to return to the canonical read; and `getCoachContext().day_read` now prefers the cached read (agentic sentence + steer) over the bare deterministic `dayRead`, so chat/coach/meal prompts echo the Brief the athlete is actually looking at. Migration **v26** (`directive-feedback-memory`) adds `health_directives.directive_key` plus trigger snapshot/status metadata so Done/Dismiss becomes agentic feedback memory: handled advice stays quiet for the same marker result, dismissed advice is not repeated unless the marker materially changes. Migration **v25** (`insights-next-step`) adds `insights.next_step` — the quiet insight's optional concrete suggestion, split out of the `rationale` blob so the Today card can render it as its own scannable line (with the reasoning tucked behind a "why this" disclosure). Migration **v24** (`garmin-strength-reconciliation`) adds `garmin_activities.exercise_sets_json` (detected strength sets) + `garmin_activities.session_id` (link to the reconciled Cairn session) and `sessions.garmin_json` (the strength physiology blob: HR/zones/calories/training-effect + the agent's one-line read + an `extrapolated` flag) — see "Garmin strength reconciliation" below. Migrations **v22** (`garmin-daily-full-dataset`) + **v23** (`garmin-activity-detail`) expand Garmin into a full-body dataset: `garmin_daily_metrics` gains ~37 columns (sleep architecture deep/light/REM/awake + restlessness, HR max/min + 7-day avg, `hrv_status`, `stress_max`, body-battery charged/drained, respiration, SpO₂, `skin_temp_dev_c`, calorie breakdown, floors, intensity minutes, `vo2max`/`training_readiness`/`training_status`/`acute_load`/`fitness_age`, and full body composition) and `garmin_activities` gains ~15 (moving time, elevation loss, aerobic/anaerobic TE + label, cadence, power, speed, ambient temp, activity `vo2max`, and `hr_zones_json` HR-time-in-zone). The sync (`src/garmin.ts`) mines the rich `getSleepData`/`getDailyWeightData` payloads it already fetched plus the connector's internal endpoints (`/usersummary-service`, `/hrv-service`, `/metrics-service/...trainingreadiness|maxmet|trainingstatus`, `/activity-service/...hrTimeInZones`), each field best-effort/null-safe in its own try/catch. `upsertGarminDailyMetric` builds its INSERT from `GARMIN_DAILY_COLS` and COALESCE-merges on conflict (a sparse re-sync never nulls a richer earlier value). Signals flow through `getGarminCoachSummary`/`getRecoverySummary` → `renderConnectedBrain` (prompt) + the PWA Brain recovery view; `GARMIN_HR_ZONE_LIMIT` (default 20) bounds per-activity zone calls. See `docs/GARMIN.md`. The HealthOS vision build added five **new tables** (no migration needed — `CREATE TABLE IF NOT EXISTS` only): `checkins` (optional subjective morning mood/energy/sleep-feel/soreness on a 1–5 scale), `daily_metrics` (source-agnostic wearable rows — Apple Health via Shortcuts etc., parallel to `garmin_daily_metrics`, `UNIQUE(source,date)`), `family_members` (the roster the coach plans around), `health_directives` (the connected-brain cross-domain directives), and `insights` (quiet one-at-a-time cross-domain reads). It added four **column** migrations: **v19** (`sessions-autoregulation`) adds `sessions.soreness`/`sessions.performance`/`sessions.joint_pain` (1-tap autoregulation feedback); **v20** (`profile-about-me`) adds `profile.about_me` (rich free-text understanding); **v21** (`health-directives-marker-uncertain`) adds `health_directives.marker`/`health_directives.uncertain`; **v26** adds `health_directives.directive_key`/`status_at`/`trigger_value`/`trigger_side`/`trigger_date`/`resurfaced_from_id`.

## Architecture: two protocol surfaces over one repo layer

The most important structural fact: **`src/api.ts` (REST) and `src/mcp.ts` (MCP) are both thin adapters over the same `src/repo.ts`.** They expose largely parallel operations. Any business-logic change belongs in `repo.ts` (or `prompt.ts`); the two surfaces should stay near-mirror wrappers. When you add a capability, expect to touch both `api.ts` and `mcp.ts` plus the underlying `repo.ts` function. Multi-step **agentic** operations that both surfaces run (session-suggest, nutrition check-in, meal swap, recipe, health review, insight generate) live in **`src/coachOps.ts`** — each exports one function returning a plain `{ok, …}` result; `api.ts` wraps it in `res.json(...)`, `mcp.ts` in `asText(...)`. The shared `runChosen(agent, prompt)` (explicit agent, or `"auto"`/blank → configured rotation with fallthrough) also lives there. Put new agentic orchestration in `coachOps.ts`, not duplicated in the adapters.

- `src/repo.ts` — all data access and domain logic: plan/sessions/sets, plan editing (`savePlanDay`/`deletePlanDay`/`replacePlan`), session-by-date lookup (`getSessionByDate`) and last-set prefill (`getLastSet`), session finish + summary (`finishSession`/`sessionSummary`), Epley est-1RM (`getProgress`) and PR detection (in `logSetByName`, returns `pr`/`est_1rm` — for `mode:'timed'` exercises a PR is a new max `duration_sec` and `est_1rm` stays null), exercise CRUD (`listExercises`/`upsertExercise`/`updateExercise` — `mode` is `'reps'` or `'timed'`), the exercise guide (`getExerciseDetail`), weekly dashboard (`getWeeklyStats` → `{ week_sessions, week_tonnage, week_sets, streak }` plus the compass fields `{ week_done, week_planned, trend_lb_wk, needed_lb_wk, pace_status, weight_lb, goal_weight_lb, goal_date }` — `week_done` is distinct training days since Monday vs `week_planned` plan days; `trend_lb_wk` is a least-squares slope over ≤21 days of weigh-ins (null under 2 points / 3 days span); `pace_status` is `'on'|'behind'|'fast'|null` where `'fast'` = losing >~1% bodyweight/wk; `week_sets` counts ALL sets incl. timed, tonnage still excludes them), bodyweight log (`logWeight`/`listWeight`, syncs `profile.weight_lb`), the TDEE + lean-safe goal feasibility math (`computeGoalCheck`), free-text activity parsing (`parseActivity`), proposals, meal plans (`getMealPlan`/`updateMealPlanDays` — replaces `parsed.days` for reorder/edit while PRESERVING all other parsed keys like `daily_kcal`/`shopping`; Free-text `addActivity`/`addFoodNote` and uploaded health docs set `enrichment_status` and (when enrichment is enabled) enqueue a background job via a lazy `import("./enrich.js")` — see the enrichment section below. Note: `plan_items` has `warmup_sets` and `target_seconds` columns (the latter prescribes hold time for timed exercises); `exercises` has a `mode` column (`'reps'` default | `'timed'`, e.g. plank/dead hang); `logged_sets` has a `duration_sec` column (timed sets may have null weight/reps); `settings` has `onboarded`, `enrich_enabled`, `art_enabled`, `art_enabled_at` (UTC stamp written by `setSettings` on every off→on flip of `art_enabled`; the spend-telemetry window — migration v15) and `meal_prefs` columns (`meal_prefs` is free-text meal/schedule preferences — e.g. "I train fasted most mornings" — embedded verbatim in the meal-plan and meal-swap prompts; trimmed, capped at 2000 chars; added in migration v11); `activities` and `food_notes` have an `enrichment_status` column; `chat_messages` has an `archived_at` column (migration v17 — set by chat reset/clear, filtered out of `listChatMessages`); `sessions` has `soreness`/`performance`/`joint_pain` (migration v19, autoregulation); `profile` has `about_me` (migration v20); `health_directives` has `marker`/`uncertain` (migration v21); `health_documents`, `context_events`, `checkins`, `daily_metrics`, `family_members`, `health_directives` and `insights` are tables added via `CREATE TABLE IF NOT EXISTS` (new tables need no migration; only ALTER TABLE column adds do).

- `src/mcp.ts` — MCP tool definitions (zod-validated). **Stateless**: a fresh `McpServer` + transport is built per POST and torn down on response close. Exposes: all existing tools plus `get_volume`, `get_calendar`, `get_session` (by date), `get_last_set`, `update_memory`, `delete_memory`, `list_exercises`/`upsert_exercise`/`update_exercise` (incl. `mode`) and `get_weekly_stats` (`log_set` also takes `duration_sec`/`exercise_mode`, `update_target` takes `target_seconds`, for timed work), `list_context_events`/`add_context_event`/`update_context_event`/`delete_context_event` (trips/injuries/life events), meal-plan editing (`get_meal_plan`, `swap_meal` — agentic single-meal swap with an optional free-text `hint`, via the shared `runChosen`; - `src/api.ts` — Express router mounted at `/api`. Exposes: all existing endpoints plus `GET /api/volume`, `GET /api/calendar`, exercise CRUD (`GET /api/exercises`, `POST /api/exercises` upsert-by-name `{name, muscle_group?, mode?}`, `PUT /api/exercises/:id` for `mode`/`muscle_group`/`cues`/`constraint_note`), `GET /api/art?kind=food|exercise|activity&q=` (cached generated artwork: the PNG with immutable caching on hit, `204` on miss/unconfigured while generation queues in the background — see `src/art.ts`), `GET /api/sessions?date=YYYY-MM-DD` (single session or `null`), `GET /api/last-set?exercise=` (prefill, or `null`), `GET /api/activities/:id` & `GET /api/food-notes/:id` (single rows, for enrichment polling), `PUT`/`DELETE /api/memory/:id`, health docs (`POST /api/health-docs` — base64-in-JSON upload, NOT multipart; responds `{ok:true, plan, meal, agent, tried}` or `{ok:false, error, tried}` with status 200 — `ok:false` is the designed failure signal; `PUT /api/meal-plans/:id/days` `{days}` — manual reorder/edit preserving the other parsed keys; `POST /api/meal-plans/:id/recipe` `{day, meal_index, agent?, force?}` — agentic recipe for one planned meal, cached on the meal in `parsed_json`: 404 only on unknown plan id; cached + `!force` → `{ok:true, recipe, cached:true}`; Uploads land in `data/uploads/` (UUID filenames, mime restricted to a concrete raster-image/PDF allowlist — no SVG; the body limit is raised to 25mb). The `?date=`/`last-set` lookups intentionally return `200 + null` on absence (not 404): the PWA's `api()` helper resolves to the body regardless of status, so a 404 error-object would read as a truthy hit.
- `src/enrich.ts` — **background enrichment engine**. See the enrichment section below.
- `src/art.ts` — **generated artwork service**: photoreal/stylized images for foods, exercises and activities via Google's `gemini-3.1-flash-image` (override with `GEMINI_IMAGE_MODEL`; needs `GEMINI_API_KEY`), cached in `data/art/` as `sha1(kind:normalized text).png`. A strictly serial in-process queue (mirrors the enrich.ts pattern: in-flight dedup by cache key, a throwing job never breaks the drain loop, plus an in-memory negative cache so failed keys aren't retried until restart) generates on cache miss while `GET /api/art` returns 204 immediately; it's a **direct REST call** (global fetch, 60s AbortController timeout, atomic tmp-then-rename write) — NOT an agents.json CLI run. Degrades gracefully: no key / `settings.art_enabled` off / known-failed → 204, nothing runs. **Semantic cache layer**: before any image call, `resolveConcept()` runs one cheap Gemini *text* call (`GEMINI_TEXT_MODEL`, default `gemini-3.1-flash`, 20s timeout, temperature 0) asking whether the query would render essentially the same picture as an existing `art_assets` row (comparison window: 150 most recent per kind; strict for exercise/activity — different movement/equipment is not a match; lenient for food — same dish reworded is) and, if not, what canonical phrase to file the new image under; the verdict persists in `art_aliases` (normalized query → asset key), so each unique phrase pays at most one text call ever and `cachedArtPath`/`requestArt` resolve through aliases on every later hit. Any canonicalize failure falls back to generating under the query's own key. **Spend telemetry**: every paid call (and every avoided generation) is recorded via `repo.recordArtUsage` into `art_usage` — actions `generate` (flat `ART_IMAGE_COST_USD`, default $0.039/image), `canonicalize` (token-priced via `usageMetadata` at `ART_TEXT_IN_USD_PER_M`/`ART_TEXT_OUT_USD_PER_M`), `reuse` (carries `est_saved_usd`), `fail`; surfaced by `GET /api/art/stats` / MCP `get_art_stats` with a since-`art_enabled_at` window plus all-time. `warmArt()` pre-queues every image the PWA will ask for (exercise names; the current non-discarded meal plan + draft's meals; ~30 recent food notes; distinct recent activity types via a duplicated `ACT_ART_PHRASE` map that MUST stay in sync with `public/app.js`) — queries are built exactly like the PWA's (`.trim().slice(0,120)`) so cache keys match; it runs ~5s after boot in `server.ts` and on `POST /api/art/warm`, returning `{queued, skipped}` (`requestArt` makes it a safe no-op when generation is unavailable). `data/art/` is a regenerable cache (safe to exclude from backups). Set `GEMINI_API_KEY` in the container environment, `.env`, or Settings if generated artwork should run; otherwise the PWA stays on SVG fallback.

## The agentic coaching loop (propose → review → apply)

Coaching never auto-applies. The flow:

1. `src/prompt.ts` builds a prompt embedding the full `getCoachContext()` snapshot and a strict JSON output contract (plus non-negotiable training/nutrition guardrails — conservative load progression, injury `constraint_note`s, lean-safe deficits). `buildMealPlanPrompt` is framed as a longevity/getting-lean coach: a shared `LONGEVITY_GUARDRAILS` block (protein anchor ~0.7–1 g/lb, 30g+ fiber, whole foods, oily fish 2–3×/wk, lean-safe only), flexible meal-slot timing rules (no pre-workout meal for fasted morning training — substantial post-training breakfast instead), the training split, and `settings.meal_prefs` verbatim. `buildMealSwapPrompt({plan, day, mealIndex, hint})` drives the single-meal swap (replacement within ±10% kcal/protein unless the hint overrides; bare single-meal JSON contract). **Vision-build prompt builders, all bound to the constitution** (calm, suggestion-not-a-gate, no 0-100 scores, pull-never-push, health findings are informational not medical advice): `buildDayReadPrompt` (the Brief — reads the whole picture, honors an athlete override, protects earned rest), `buildSessionPrompt` (on-demand session honoring time/equipment/injury + the day read), `buildNutritionCheckinPrompt` (adherence-neutral adaptive check-in over the derived expenditure — proposes a small ±100–250 kcal change only on genuine drift, protein-first floors, `change:false` the common answer), `buildInsightPrompt` (ONE genuine cross-domain connection or `{found:false}`, deduped against `recentInsightTexts`) and `buildWeeklyReadPrompt` (a standing "how the week went + the one change" stored as a `weekly_read` insight). **The connected brain is rendered into every plan-shaping prompt** by the shared `renderConnectedBrain(ctx, {domains})` helper, which folds the active cross-domain `directives` (filtered to the relevant domain — nutrition for meals, training/watch for the coach/day-read) plus the unified `recovery` view into a compact plain-language block, so a flagged lab already shapes meals & training (returns `""` when there's nothing to say). `CONTEXT_GUARDRAILS` (shared across the coach/chat/meal prompts) now also covers FAMILY (plan around commitments, let `about_me` personalize tone) and HEALTH DIRECTIVES (fold nutrition/training directives in, respect `watch` items; an `uncertain`/uncited directive is a softer nudge). `buildHealthReviewPrompt`'s `HEALTH_REVIEW_SCHEMA` now emits a `directives[]` array (persisted via `applyReviewDirectives`). `buildChatPrompt` adds a **PROGRESSIVE UNDERSTANDING** guardrail: when the DATA shows an obvious gap (no `about_me`, an unknown training-time / food like-dislike) the agent MAY ask ONE brief low-friction question and emit an `add_memory` action for the answer — never a questionnaire. Chat safe-actions (applied immediately) now also include `log_food`, `log_health` (extracts every marker into Health records) and `add_context_event` (trips/injuries/life events/family commitments).
2. `src/agents.ts` `runAgent()` spawns an **external CLI subprocess** (Claude Code, Codex, Antigravity, Grok, or `stub`) defined in `agents.json`, substituting `{prompt}` into the args. `extractJson()` pulls the JSON object out of the CLI's stdout.
3. The result is stored as a `draft` row (`plan_proposals` or `meal_plans`) — never applied directly.
4. `applyProposal()` walks `parsed.changes[]` and calls `updateTarget()` per change, flipping status to `applied`. (It also recognizes two newer shapes: a `parsed.days` payload routes to `replacePlan()` for a full restructure, and a `kind:'nutrition_target'` payload from the adaptive nutrition check-in is advisory — there's no plan to mutate, so "applying" it is a clean acknowledgement on every surface, surfaced via the Energy Balance card, not the plan-proposals apply button.)

**Agent selection.** Callers either name an agent or pass `agent: "auto"` / omit it. Auto uses `repo.pickAgentOrder()` (strategy from the `settings` table: `round_robin` default / `random` / `priority`, over the enabled agents in their configured order) and `agents.runAgentWithFallback()`, which tries agents in order until one returns parseable JSON — so a dead CLI login or timeout transparently falls through to the next. `api.ts` and `mcp.ts` share a `runChosen()` helper for this. `agents.json` still defines each backend (command, args, stdin-vs-arg input, env requirements); which are **enabled/ordered** and the **strategy** live in `settings` and are editable at runtime (PWA Settings tab, `/api/settings`, or `get_settings`/`set_settings` MCP tools) — no redeploy.

`src/scheduler.ts` is an optional weekly auto-draft: a 60s `setInterval` reads `settings` each tick (so the schedule is toggleable live) and, when `coach_enabled` and the configured `coach_day`/`coach_hour` match, drafts one proposal via the auto rotation and leaves it for manual apply. First-run settings seed from the legacy `COACH_AGENT`/`COACH_DAY`/`COACH_HOUR` env vars.

**Conversational chat** (`POST /api/chat`, `buildChatPrompt`) is the same loop with a richer contract: the agent returns `{ reply, actions[] }`. The API applies *safe* actions immediately (`log_activity`/`log_set`/`set_profile`/`add_memory`) but turns plan-affecting actions into **draft proposals** — `plan_update` (target tweaks → `parsed.changes`) and `plan_restructure` (frequency/split change → `parsed.days`). `applyProposal()` handles both shapes: a `parsed.days` payload routes to `replacePlan()` (full restructure), otherwise it loops `updateTarget()` over `parsed.changes`. Chat turns persist in `chat_messages`. The Chat tab groups messages under day dividers with older days collapsed behind a "Show earlier" affordance, offers starter chips when the conversation is empty, and has a header "fresh start" button → `POST /api/chat/reset` (distill-to-memory, then archive).

## Background enrichment (`src/enrich.ts`)

Free-text activity logs and food notes save **instantly** via the offline regex parser (`parseActivity`), then — if `settings.enrich_enabled` (default on) and an agent is reachable — a **background, in-process serial queue** runs `buildEnrichPrompt()` through `runAgentWithFallback()` to (a) refine the entry's structured fields and (b) distill genuinely-notable durable facts into the `memory` table. **Uploaded health documents** (bloodwork/DEXA/other) are the third enrichment kind: the job hands the agent the **absolute file path** (`buildHealthEnrichPrompt`) — the CLIs (Claude Code/Codex) can open local files — to extract markers + a summary + memory. It degrades gracefully end-to-end: disabled / no agent → `skipped` (written directly at insert, no queue churn); a failure or wrong-shape (e.g. coach-proposal) response → `failed`/no-op with the regex parse (or the as-uploaded doc) left intact.

- **Kinds**: `'activity' | 'food' | 'health'`. **Status machine** on `activities`/`food_notes`/`health_documents` `.enrichment_status`: `pending` → `in_progress` (set *before* the agent `await`, so a crash leaves a recoverable marker) → `done`/`failed`/`skipped` (or `null` = not applicable). `recoverPendingEnrich()` (called from `server.ts` at boot) re-enqueues `pending`+`in_progress` rows for all three kinds.
- The queue is **strictly serial** (one CLI subprocess at a time), uses a shorter 120s timeout than the coach loop, and a throwing job never escapes the drain loop. A health job refuses a non-absolute `file_path` (skips) — that's the only thing constraining the agent's file read to uploads.
- Agent-provided fields are **type-coerced and clamped** before write; `memory` items are added via `addMemory` (which dedupes exact repeats). Prompt-only dedup is a soft hint, so the code-level dedup is the real guard.
- The frontend **polls** `GET /api/activities/:id` / `/food-notes/:id` / `/health-docs/:id` to watch the status flip and upgrade the entry in place.

**Personal-context layer.** `health_documents` (uploaded labs/scans → analyzed markers + summary) and `context_events` (trips / injuries / life events, with dates + a `meta_json` for kind-specific fields) both feed `getCoachContext()`. A shared `CONTEXT_GUARDRAILS` block in `buildCoachPrompt`/`buildChatPrompt`/`buildMealPlanPrompt` directs the coach to plan **around** them: travel-friendly/deload weeks over trip dates, de-load/swap exercises hitting an injured area, dial volume back during flagged life events, and factor health flags (e.g. low ferritin) — explicitly framed as informational, not medical advice.

## Domain gotchas

- **Weight encoding**: negative `weight`/`target_weight` = assisted movement (e.g. `-30` = 30 lb assist); `null` = bodyweight. The PWA and prompts rely on this.
- **Timed exercises** (`exercises.mode = 'timed'`): sets log `duration_sec` (weight/reps may be null), plans prescribe `target_seconds`, and progression is in seconds — never load. Proposal `changes[]` may carry `target_seconds` instead of `target_weight`.
- Est-1RM is Epley on the best set per day.
- Sessions are keyed by date — `getOrCreateSession` reuses today's session, so logged sets accumulate into one session per day.
- **Garmin strength reconciliation.** A synced Garmin *strength* activity (`isStrengthGarminType`) is modeled as a Cairn **session**, not a loose activity — `upsertGarminActivity` deliberately skips the generic `activities` row for it (no RECENT duplicate). `src/garmin.ts` also pulls Garmin's detected `exerciseSets` (`fetchExerciseSets`, bounded by `GARMIN_STRENGTH_LIMIT`). After each sync (and via `POST /api/garmin/reconcile` / MCP `reconcile_garmin_strength`), `repo.reconcileGarminStrength()` runs the **deterministic** merge — attaches the physiology blob (HR/zones/calories/TE) to `sessions.garmin_json`, links `garmin_activities.session_id`, deletes any stale generic row — *always*, even with no agent. Then the **agentic** layer runs on the serial enrichment queue (`enrich.ts` kind `'garmin_strength'` → `buildGarminStrengthPrompt`): it writes the one-line "body's reaction" summary and **logs the exercises Garmin detected that aren't already in the session** via `logSetByName` (a code-level guard skips any exercise that already has logged sets, so hand-logged sets are never overwritten/duplicated and re-syncs stay idempotent). Degrades cleanly: enrichment off / no agent → physiology merges, no narrative, no extrapolation. Surfaced as the "Garmin · body's reaction" card on Today (`garminSessionCard`).
- **Autoregulation feedback** (`sessions.soreness`/`performance`/`joint_pain`, set via `setSessionFeedback`) is optional 1-tap signal that INFORMS coach selection (bend volume, de-load a sore joint) — it NEVER overrides progressive overload or auto-changes the plan. Likewise the subjective `checkins` are a parallel day-read signal and the graceful-degradation path when there's no wearable; absent signals never force a rest read.
- **The connected brain (markers → directives propagation).** A flagged/sub-optimal lab marker propagates into the domains it touches (nutrition/training/watch) as `health_directives` via the deterministic `deriveDirectives()` engine and `MARKER_MAPPINGS`. Two sources coexist and each clears/rewrites only its own rows: `'markers'` (the deterministic engine) and `'health_review'` (agent-emitted on a saved review). Directives are INFORMATIONAL, not medical advice; a directive flagged `uncertain` (or lacking a `citation`) is a softer nudge, not a hard rule. Nothing auto-applies — the user flips `active|resolved|dismissed`.
- **Optimal-zone vs lab-range.** `OPTIMAL_ZONES` are evidence-anchored longevity/preventive-cardiology target bands (ApoB, LDL, HbA1c, ferritin, vitamin D, …), DISTINCT from the lab's population reference interval — a value can read "in range" yet sit far from optimal. `prioritizeMarkers()` ranks by an INTERNAL `impact_score` (distance-from-optimal, weighted by whether there's an actionable lever, plus a flag floor); **never surface that score** — the constitution bans 0-100 grades. The UI shows in/out-of-optimal and the direction, never the number. Markers are also bucketed into meaningful health groups (`MARKER_GROUPS`, longest-match-wins on the name, same matcher shape as `matchOptimalZone`) and carry a deterministic `trend` (`{dir:'rising'|'falling'|'stable'|null, change, span_days, n}` — stable when the net change is small vs the series' own spread); `getMarkerHistory()` and `prioritizeMarkers()` both return `{ markers, groups }` (canonical-ordered `groups` present) with per-marker `group`/`group_label`/`trend`. The review prompt surfaces each priority marker's group + trend direction + reading recency so the coach speaks to trajectory ("rising over ~14 mo"), not just the latest value. Reading recency renders human via `relAge()`/`absDate()` (see `docs/DESIGN.md`), never a bare month-year.
- **Product constraints (from `docs/VISION.md`, enforced in prompts AND UI).** The Brief and every read are a SUGGESTION, never a gate or verdict — the athlete drives. NO numeric scores anywhere (no 0-100 grades, no metric walls). Insights and the weekly read are PULL, never push — they wait in-app, never a notification or nag; the generator emits ONE genuine connection or `{found:false}` and dedupes against what it already said. Adaptive nutrition is adherence-NEUTRAL: a thin logging week lowers confidence, never blames; `change:false` is the calm, common answer.
- **No authentication by default** (single-user, trusted-network model). An OPTIONAL shared token gates `/api` and `/mcp` when `CAIRN_AUTH_TOKEN` is set — implemented as one global guard in `src/auth.ts` (wired in `src/server.ts` before the routers; exempts `GET /api/health` and the static PWA shell). Unset = behaves exactly as before. The PWA sends it as `X-Cairn-Token` (or `?token=` on direct resource URLs via the `withToken()` helper); MCP/API clients use `Authorization: Bearer`. Keep it on localhost, a LAN, Tailscale/VPN, or another trusted private network regardless. The `cairn-home` Docker volume holds live OAuth tokens for the CLI logins. `src/server.ts` also sets conservative security headers (CSP/nosniff/frame-deny, `x-powered-by` off) and scopes the 25mb body limit to the health-doc upload route only (1mb elsewhere). The container runs its main process as a non-root `app` user (an entrypoint fixes volume ownership then drops privileges); one-off `docker compose exec` commands that must persist a login need `-u app`.

## PWA

`public/` is dependency-free vanilla JS. `public/app.js` is a single-file tabbed client (Today / Plan / Progress / Chat / Me / Settings) that talks only to `/api/*`; `sw.js` is a minimal cache-first service worker — **any change to files under `public/` MUST bump the `CACHE` version constant at the top of `sw.js` in the same commit** (currently `cairn-v48`), or installed PWA clients keep serving the stale assets forever. No build step touches `public/`. The visual language is the **"Atelier" design system** — warm-cream editorial theme (Fraunces display + Schibsted Grotesk UI, terracotta/sage accents) — whose frozen contract (palette, class names, motion rules) lives in `docs/DESIGN.md`; read it before touching `styles.css` or view markup. `public/art.js` is the **CairnArt SVG illustration library** (plain script loaded before `app.js`): `CairnArt.food(text)` / `.exercise(name, muscleGroup)` / `.activity(type)` keyword-map inputs to hand-drawn studio-style illustrations and return static SVG strings — caller text is never interpolated into the SVG, so its output is inserted into `innerHTML` raw (everything else still goes through `escHtml`/`escAttr`); app.js wraps every call in the guarded `art()` helper so a missing art.js degrades to text-only. Notable surfaces: **Today** now opens to the day-read **Brief** (`/api/today-read` → a calm `kind`/`headline`/`why`/`focus` read with override chips like "rough night" that re-fetch with `?override=`, and a "build me a session" launchpad → `/api/session-suggest` — a suggestion, never a gate), plus effortless capture (one-tap **frequents** from `/api/frequent-foods`, **voice** input via the Web Speech API `SpeechRecognition`/`webkitSpeechRecognition` when supported, an optional morning **check-in** card → `/api/checkins`), a quiet **insight card** surfacing the first of `/api/insights` with thumbs up/down (`PUT /api/insights/:id`) and optional gated `/api/insights/generate`, post-session 1-tap **autoregulation feedback** → `/api/sessions/:date/feedback`; it keeps the date picker, set-by-set logging with prefill + `N / M` progress, "+ Add exercise", and the bodyweight quick-add chip. **Progress** holds History/1RM/Volume/Weight/Calendar plus an **Energy Balance** view (`/api/nutrition/expenditure`, with an optional adaptive **nutrition check-in** → `/api/nutrition/checkin`). **Me** has a Profile / Memory / Health / Life / **Family** sub-nav: Profile now includes the free-text **about_me** field (`PUT /api/profile`), Memory is the curate-able coach memory, **Health** keeps the uploaded-docs + **marker-trends** views (markers grouped into meaningful health groups — Lipids & Cardiovascular, Metabolic & Glucose, Iron & Red Blood, … — each row tap-expands into an inline progress chart with the optimal-zone band shaded, the trend in plain words, and human-friendly recency like "3 months ago"; the tab is fed by `/api/markers/priority`, the optimal-band superset) AND a **Brain** view (recovery from `/api/recovery`, optimal-zone **priority markers** from `/api/markers/priority`, and grouped **directives** from `/api/directives` with status flips and a re-derive button → `/api/directives/derive`), **Life** is the trips/injuries/events timeline (with a subtle active-context banner mirrored on Today), and **Family** is the CRUD roster (`/api/family`). **Settings** adds an **Apple Health** connect card (the iOS Shortcut posts daily metrics to `/api/health-metrics`). Enrichment polling guards against stale-tab writes via a `pollToken` bumped on every (re-)render. All server-supplied strings rendered into `innerHTML` must go through `escHtml`/`escAttr`.
